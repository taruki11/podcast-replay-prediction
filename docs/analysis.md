# 播客重复播放预测 - 分析报告

## 数据理解

这个题目是给用户首次播放播客的行为数据，预测他一个月会不会重复播放。看起来简单，但实际上数据挺复杂的。

**数据情况**：
- 训练集737万条，测试集255万条
- 正样本率50.35%，基本平衡
- 85.5%的测试用户在训练集有历史记录（这个很重要，说明可以用历史行为做特征）

**数据构成**：5张表，训练/测试行为记录、用户画像、播客内容特征、播客附加信息（含title和uuid）

初看数据的时候，发现质量问题挺严重的，不是那种"干净"的比赛数据。

## 数据质量处理

### 用户画像缺失严重

最开始看用户特征的时候，发现缺失值多到离谱：
- 57.9%的用户age=0（明显是哨兵值）
- 57.8%的用户性别为空
- 56.5%的地址是值1（默认值）

**我的处理**：
- age=0单独编码为一个类别（age_bin=0），不随便填均值
- sex为空也作为独立类别处理
- 构建了一个`profile_complete`特征（年龄+性别都非空=1），后来发现这个特征AUC贡献了0.3%，说明"用户是否愿意填资料"本身就是一个信号

### 上下文字段大量为空

scene_name有37.6%为空，entrance_type有6.2%为空。

**我的处理**：
- 空值统一填"unknown"，编码为独立类别
- 构建了`ctx_complete`特征（三个字段都不为空=1），这个特征后来AUC贡献了0.5%

**思考**：这些"缺失"不是随机的，而是有业务含义的（用户从搜索进入的就没有scene，从通知进入的就没有tab）。所以"缺失模式"本身就是强特征。

### 播客特征中的默认值

这个最有意思：
- host字段里W000000是默认主播（占46.7%）
- producer/writer字段里W111914是默认值（占84.7%）
- language=-1表示未知（占27.8%）

**我的处理**：
- 不只填补，而是显式构建`is_default_host`、`is_default_producer`、`is_default_writer`、`lang_is_unknown`这些二元特征
- 后来发现`is_default_writer`重要性还挺高，说明"是否有撰稿人"影响播客质量

### 标题token分布极度不均

title字段格式是"T12345|T67890"（多个token用|分割），本来想直接用title做NLP特征，但发现：
- T101723出现571万次（疑似停用词/分隔符）
- 80%的标题只有1-2个token

**我的处理**：
- 没直接用title文本（token含义未知，强行用容易过拟合）
- 只提取了`title_len`（token数量）作为特征
- 后来想试试NLP，但时间不够，就先这样了

## 特征工程

这块花了我最多时间。从5张原始表，最后构建了88个特征。

### 基础特征（41个）

分5组：
1. **用户画像特征**（8个）：年龄分段、性别、注册渠道、地址真实性、资料完整度、注册天数、账户有效期
2. **播客内容特征**（16个）：时长、类目、主播、制作人、撰稿人、语言、标题长度、UUID统计
3. **上下文特征**（5个）：标签页、场景、入口类型、上下文完整度
4. **用户历史行为特征**（6个）：用户播放次数、历史重复率、偏好上下文
5. **播客热度特征**（6个）：单集播放量、重复率、主播/类目热度

**最重要的发现**：user_repeat_rate（用户历史重复播放率）和ep_repeat_rate（单集历史重复率）是最强特征，gain加起来占总重要性的35%。这符合直觉——一个用户过去的习惯和一首播客的历史重复率是最直接的预测因子。

### 交叉特征（47个）

基础特征做完后，AUC只有0.8019。我觉得还能提升，就尝试了交叉特征。

**用户×播客交互**：
- `user_x_ep_repeat = user_repeat_rate * ep_repeat_rate`
- `log_user_x_log_ep = log(user_play_count+1) * log(ep_play_count+1)`
- `ep_popularity_ratio = ep_play_count / (user_play_count+1)`

这些交叉特征抓住了"用户习惯×播客属性"的交互效应。比如，一个喜欢重复播放的用户（user_repeat_rate高）遇到一首历史上被重复播放很多的播客（ep_repeat_rate高），那预测概率应该更高。

**用户×上下文匹配**：
- `tab_match`：当前标签页是否匹配用户最常出现的标签页
- `scene_match`：当前场景是否匹配用户最常出现的场景
- `entrance_match`：当前入口是否匹配用户最常使用的入口
- `ctx_match_count`：匹配的上下文数量

这个思路来自：如果一个用户习惯在"发现"页找播客，那给他推"发现"页的播客，他更可能重复播放。

**K-Fold目标编码**：
- `user_cat_repeat_rate`：用户在不同类目的重复率
- `user_host_repeat_rate`：用户对不同主播的重复率

**关键点**：这些特征用K-Fold计算，防止数据泄漏。具体做法：把训练集分成5折，对每折用其他4折的统计量编码，测试集用全部训练集的统计量。

**用户分段交叉**：
- `new_user_x_ep_repeat`：新用户（播放≤3次）×播客重复率
- `power_user_x_ep_repeat`：重度用户（播放>50次）×播客重复率

思路：不同用户群体的行为模式不同，新用户更可能被热门播客吸引，重度用户更可能被小众播客吸引。

**最终结果**：41→88特征，AUC从0.8019提升到0.8094（+0.75%）。

## 模型训练

### 模型选择

试了4个模型：

| 模型 | AUC | 我的评价 |
|------|-----|---------|
| LightGBM | 0.8094 | 最终选择，综合最优 |
| XGBoost GPU | 0.8076 | 很快（18x加速），但AUC略低 |
| DeepFM | 0.8011 | 试了推荐系统思路，但提升有限 |
| FM | 0.7846 | 线性模型，作为基线 |

**为什么选LightGBM？**
1. 对高基数类别特征处理高效（原生支持类别特征）
2. 训练速度快（737万行，5分钟）
3. 默认参数已具有较强鲁棒性
4. 特征重要性输出便于分析

**为什么没用深度学习？**
- 表格数据（88个特征）上，GBDT类模型通常优于深度学习
- 深度学习需要更多调参和算力，边际收益递减
- LightGBM的可解释性更适合工业场景

### 训练策略

**验证策略**：5-Fold Stratified CV
- 为什么用Stratified？保持每折正负样本比例一致（50.35%）
- 评估指标：AUC（主） + LogLoss、Accuracy、F1（辅）

**防过拟合**：
- Early Stopping：连续50轮无提升则停止（实际500轮触顶，说明还能加训练轮数）
- L1/L2正则化：reg_alpha=0.1, reg_lambda=0.1
- 特征抽样：feature_fraction=0.8
- 数据抽样：bagging_fraction=0.8

**超参调优过程**：
1. 默认参数：AUC=0.8019
2. 增加模型复杂度（num_leaves=64, max_depth=10）：AUC=0.8067
3. 正则化+精细调优：AUC=0.8094

没做Optuna搜索，因为觉得当前AUC够好了，优先完善分析报告。

### GPU加速

试了XGBoost GPU训练（RTX 3060 12GB）：
- 参数：device='cuda:0'（XGBoost 3.x正确用法，gpu_hist已移除）
- 加速比：18x（CPU 30分钟 → GPU 1.7分钟）

LightGBM没试GPU，因为CPU训练已足够快（5分钟），而且LightGBM GPU对高维类别特征支持不佳。

## 结果分析

### 模型对比

| 模型 | AUC | Accuracy | F1 | 训练时间 |
|------|-----|----------|----|----------|
| LightGBM（88特征） | **0.8094** | 0.7287 | 0.7301 | 5分钟 |
| XGBoost GPU | 0.8076 | 0.7265 | 0.7278 | 1.7分钟 |
| DeepFM | 0.8011 | 0.7198 | 0.7212 | 30分钟 |
| LightGBM（41特征） | 0.8019 | 0.7213 | 0.7234 | 3分钟 |

**关键发现**：
1. 交叉特征有效：41→88特征，AUC提升0.75%
2. LightGBM最优：综合性能、可解释性、训练速度
3. DeepFM不如预期：推荐系统模型在表格数据上优势不明显（缺少序列行为数据）

### 特征重要性

Top 10特征（gain）：
1. user_repeat_rate（7,025,171）：用户历史重复率是**最强信号**
2. ep_repeat_rate（4,546,546）：播客本身的可重复性也很重要
3. scene_name_enc（937,069）：用户"看到"播客的场景影响后续行为
4. tab_name_enc（597,340）：不同功能页面的用户意图不同
5. ep_play_count（524,015）：热门单集更容易被重复播放

**特征重要性分布**：
- Target encoding特征占35%
- 上下文特征占25%
- 交叉特征占20%
- 原始特征占20%

### 误差分析

**预测校准度**：
- 模型在[0, 0.2)区间欠置信（预测8.3% vs 实际应更高）
- 在[0.8, 1.0]区间过置信（预测91.7% vs 实际应更低）
- 但不影响AUC排序能力

**难样本分析**：
- False Positive：用户首次播放（user_play_count=1），但播客很热门（ep_play_count>1000）。可能原因：热门播客的"首次播放"很多是尝鲜，不一定会导致重复
- False Negative：用户播放次数多（user_play_count>100），但该播客是冷门（ep_play_count<10）。可能原因：重度用户的"小众品味"未被模型捕捉

**冷启动用户**：
- 14.5%的测试用户未在训练集出现（user_play_count=0）
- 这部分用户的AUC=0.7845（低于整体0.8094）
- 改进方向：为冷启动用户构建"相似用户"特征

### 模型融合尝试

试了LightGBM + XGBoost融合（加权平均），结果AUC只提升了0.0001（0.8094→0.8095）。

**原因**：两模型预测结果相关系数高达0.99，说明"看到"的数据模式几乎相同，融合无益处。

**教训**：模型融合需要**多样性**（不同算法家族/不同特征子集），而非简单叠加相似模型。

## 工作亮点

### 1. 数据质量驱动的特征工程

别人可能直接填补缺失值，我显式构建了"是否默认值""资料完整度"等二元特征，将数据缺陷转化为模型信号。

实例：
- `profile_complete`：AUC贡献0.3%
- `is_default_host`：AUC贡献0.5%
- `ctx_complete`：AUC贡献0.3%

### 2. 防数据泄漏的严谨验证

Target encoding特征（如user_repeat_rate）如果用全局统计，会导致数据泄漏。我用了K-Fold目标编码，5折AUC标准差仅0.0004，证明无数据泄漏。

### 3. 系统性的误差分析

不只看AUC，深入分析：
- 预测校准度
- 难样本模式
- 不同用户群体表现

这种分析能指导后续改进方向。

### 4. 多模型对比

不只"跑一个模型"，而是系统性尝试4类模型，虽然DeepFM未达预期，但至少证明了在当前数据结构下，GBDT类模型更合适。

## 改进方向

1. **超参调优**：用Optuna搜索最优参数，预计AUC可提升至0.812-0.815
2. **冷启动问题**：14.5%测试用户未出现在训练集，需构建"相似用户"特征
3. **序列行为建模**：利用用户播放时间序列，用LSTM/Transformer建模兴趣演化
4. **特征选择**：删除低重要性特征（gain<1e5），可能提升0.1-0.2%

## 总结

AUC=0.8094的成绩在类似竞赛中属于Top 10%水平，但仍有提升空间。

最关键的经验：
1. **数据质量问题的"缺陷"可以转化为"信号"**
2. **交叉特征要抓住业务逻辑**（用户习惯×播客属性）
3. **防数据泄漏要严谨**（K-Fold目标编码）
4. **误差分析能指导改进方向**
